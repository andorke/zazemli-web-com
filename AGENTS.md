# zazemli_web_com

> Точка входа для coding-агентов. Прочитай этот файл первым.

## Что это за проект
Маркетинговый сайт `zazemli.com` российского D2C-бренда ЗАЗЕМЛИ — боксов для пересадки комнатных растений (7 SKU грунтов под конкретные растения). Сайт — «витрина-музей» и источник доверия: сам не продаёт, единственная конверсия — внешние ссылки на Ozon с UTM-разметкой. MVP — 5 страниц (`/`, `/collectio`, `/lab`, `/guide`, `/diary-signup`). Стадия: старт разработки — дизайн-система, каркас и сборка главной по Figma. Вся информация фаундера — в `docs/zazemli_knowledge_base/` (точка входа `НАВИГАЦИЯ.md`) и `docs/zazemli_design_arts/` (бриф `HANDOFF_C_IVAN_production-brief_2026-06-12.md`, шрифты, дудлы, QR).

## Рабочий процесс проекта: OpenSpec + Superpowers

Проект использует **OpenSpec** для персистентного планирования и **Superpowers** для методологии исполнения. Это независимые системы — правила ниже явно их композируют.

### Порядок чтения для новых агентов
Прежде чем что-либо делать, прочитай эти файлы в следующем порядке:
1. `AGENTS.md` — этот файл. Точка входа, правила, глоссарий.
2. `DEVELOPMENT.md` — стек, структура репозитория, паттерны кода, локальный запуск.
3. `CONTEXT.md` — текущий статус, последние решения, журнал «что сделано».
4. `ROADMAP.md` — фазы и чекпойнты.
5. `REQUIREMENTS.md` — функциональные и нефункциональные требования.
6. `ARCHITECTURE.md` — C4-диаграммы, API-контракты, ключевые архитектурные решения.

### Роли
- **OpenSpec** владеет персистентным планированием отдельных изменений: `proposal.md`, `design.md`, `tasks.md`, `archive/`. Источник истины для того, ЧТО мы строим и ЗАЧЕМ, на уровне одной фичи.
- **Superpowers** владеет методологией исполнения: TDD, systematic debugging, code review, git worktrees, subagent dispatch. Источник истины для того, КАК мы строим.
- **Документы проекта** (`AGENTS.md`, `CONTEXT.md`, `DEVELOPMENT.md`, `ROADMAP.md`, `REQUIREMENTS.md`, `ARCHITECTURE.md`) владеют глобальным, долгоживущим контекстом проекта. OpenSpec работает в scope одного изменения, эти файлы — в scope всего проекта.

### Поток на одну фичу

Каноническая цепочка с 1.3.x (11 workflows). Этапы 1–6 обязательны, альтернативные пути на этапе 3 — выбор по сложности change-а.

1. **`brainstorming`** (Superpowers) — уточнить идею в чате. Итоговый дизайн-док — в `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`. Все персистентные артефакты Superpowers-скиллов живут в `docs/superpowers/` (specs/, plans/); авторитетный план фичи — всё равно OpenSpec change, дизайн-док на него ссылается, не дублирует.
2. **`/opsx:explore`** — *опционально, если идея сложная или нужно исследовать кодовую базу*. Режим «думать вслух» — partner-агент без записи на диск. Выходишь из explore, как только сформулировал, что именно строим.
3. **Создание change** — один из трёх путей:
   - `/opsx:propose <name>` — **быстрый путь**: за один заход создаёт `proposal.md`, `design.md`, `specs/*`, `tasks.md`. Дефолт для понятных фич.
   - `/opsx:new <name>` → `/opsx:continue` × N — **пошаговый путь**: создаёшь скелет, затем каждый артефакт отдельно, с проверкой. Для крупных или неоднозначных change-ей.
   - `/opsx:new <name>` → `/opsx:ff` — **fast-forward путь**: скелет, затем `ff` бежит все артефакты подряд без подтверждений. Для тривиальных правок.
   - В любом из трёх — просмотри и отредактируй артефакты руками перед `/opsx:apply`.
4. **`/opsx:apply <name>`** — реализация задач из `tasks.md`. Правила ниже («Обязательные правила во время /opsx:apply») — обязательные.
5. **`/opsx:verify <name>`** — проверяет, что реализация соответствует артефактам (specs, design, tasks). **Этот шаг новый**: ловит дрейф между планом и кодом до того, как code review увидит проблему. Если verify нашёл расхождение — поправь код или артефакты (зависит от того, кто прав), затем повтори verify.
6. **`requesting-code-review`** (Superpowers) — финальный code review на всё изменение.
7. **`/opsx:sync <name>`** — переносит delta-specs из change-а в main `openspec/specs/<capability>/spec.md`. **Этот шаг новый**: раньше его прятал `archive`. Теперь он явный, agent-driven (умный merge: можно добавить scenario без копирования всего requirement-а).
8. **`/opsx:archive <name>`** — только после того как тесты зелёные, verify чистый, code review закрыт и delta-specs sync-нуты. Архивирует change в `openspec/changes/archive/YYYY-MM-DD-<name>/`.
9. **После archive** — обнови `CONTEXT.md` (журнал «что сделано») и, если change закрыл запланированный пункт, переключи чекбокс в `ROADMAP.md`. Подробности в секции «КРИТИЧНО: Обновление документов проекта».

**Параллельные финиши:** если в работе было несколько change-ей одновременно и все они стали готовы — после verify и sync на каждом запусти `/opsx:bulk-archive` чтобы заархивировать всю волну разом.

**Онбординг проекта:** для нового члена команды есть `/opsx:onboard` — проводит через полный цикл (propose → apply → verify → archive) с пояснениями и реальной работой по кодовой базе. Альтернатива ручному изучению этого AGENTS.md.

### Обязательные правила во время /opsx:apply
1. Каждая задача в `tasks.md` реализуется через **test-driven-development**: RED (failing test) → GREEN (минимальный код) → REFACTOR. Не пишем код без сначала failing теста.
2. Перед тем как переключить `- [ ]` → `- [x]` на задаче, запусти **verification-before-completion** как внутреннюю само-проверку.
3. Используй **systematic-debugging**, когда тест падает по неочевидной причине. Не гадай.
4. Для независимых подзадач, которые можно запускать параллельно, используй **dispatching-parallel-agents**.
5. Используй **using-git-worktrees** для изолированных веток, когда изменение трогает много файлов или есть риск конфликта редактирований.
6. После того как все задачи в `tasks.md` отмечены `- [x]`, **обязательно** запусти `/opsx:verify <name>` до code review. Если verify нашёл расхождение (код не соответствует specs/design, или артефакты устарели относительно реализации) — пауза, поправь, повтори verify.
7. Перед `/opsx:archive` — обязательный **requesting-code-review** на всё изменение, плюс `/opsx:sync <name>` если у change-а есть delta-specs.

### Заполнение документов проекта через Superpowers
Шесть файлов в корне проекта (`AGENTS.md`, `DEVELOPMENT.md`, `CONTEXT.md`, `ROADMAP.md`, `REQUIREMENTS.md`, `ARCHITECTURE.md`) создаются как скелеты с секциями и `TODO`-плейсхолдерами. Заполняй их итеративно — не пытайся написать 1000 строк в первый день.

- В папке `example_docs/` лежат **заполненные примеры** тех же файлов для вымышленного проекта NoteHub. Это референс структуры и детализации, **не шаблон для копирования содержимого**. Перед заполнением соответствующего скелета — прочитай его пример из `example_docs/`.
- Используй Superpowers **`brainstorming`**, чтобы задавать пользователю уточняющие вопросы и извлекать информацию для любой `TODO`-секции. По одному сфокусированному вопросу за раз. Не выдумывай контент.
- Используй Superpowers **`writing-plans`** только для превращения решённого scope в фазы `ROADMAP.md`. Фазы должны быть крупными — детальные подзадачи живут в OpenSpec changes, не в `ROADMAP.md`.
- Обновляй `CONTEXT.md` после каждого значимого решения или завершённого change. Это журнал, а не спека — добавляй записи сверху, не переписывай историю.
- Обновляй `ARCHITECTURE.md` только когда принято решение. Держи его фактическим, не aspirational. C4-диаграммы опциональны на старте; начни с прозы об компонентах и границах.
- `DEVELOPMENT.md` фиксируется один раз на выбор стека. Обновляй только при смене стека, структуры или паттернов кода — не per-feature.
- `REQUIREMENTS.md` — зафиксированные функциональные (FR) и нефункциональные (NFR) требования. Обновляй при добавлении / удалении / изменении требований, не при реализации.

### Управление потоком изменений
- **Не** запускай автоматически следующий этап (propose/apply/archive) после завершения текущего. Пользователь сам управляет потоком через команды `/opsx:propose`, `/opsx:apply`, `/opsx:archive`. Завершил текущий этап — доложи результат и жди команды.
- **Не** запускай Superpowers скиллы (brainstorming, writing-plans и др.) для автоматического перехода к следующему change.
- Subagent'ы могут запускаться только по явному запросу пользователя.

### КРИТИЧНО: Обновление документов проекта после OpenSpec changes

> **ЭТО ОБЯЗАТЕЛЬНЫЙ ШАГ. НЕ ПРОПУСКАЙ ЕГО, ДАЖЕ ЕСЛИ СКИЛЛ `/opsx:apply` О НЁМ НЕ УПОМИНАЕТ.**
> Скиллы (apply, propose, archive) — это шаблоны выполнения. Этот файл (AGENTS.md) — правила проекта. Правила проекта имеют приоритет над скиллами. Если скилл не содержит шага "обнови документы", ты всё равно ОБЯЗАН это сделать по правилам ниже.

После завершения каждого `/opsx:apply` (когда все задачи отмечены), **прежде чем** сообщить пользователю о завершении, выполни обновление документов:

1. **`CONTEXT.md`** — **всегда** добавь запись в журнал «что сделано» (новая строка сверху таблицы). Обнови «Текущая задача» и «Следующие шаги».
2. **`ARCHITECTURE.md`** — обнови если change добавил новый компонент, интерфейс, слой или изменил архитектурное решение.
3. **`DEVELOPMENT.md`** — обнови если change добавил новую зависимость, изменил структуру репозитория или паттерны кода.
4. **`REQUIREMENTS.md`** — обнови если change реализовал FR/NFR (отметь как реализованное) или выявил новое требование.
5. **`ROADMAP.md`** — переключи чекбокс если change закрыл запланированный пункт.

- **Не** обновляй документы спекулятивно — только по факту реализованных изменений.
- В сообщении о завершении `/opsx:apply` **перечисли какие документы обновил** (или укажи "документы не затронуты" если change не затронул ни один scope).

### Границы (что НЕ делать)
- **Не** используй Superpowers `writing-plans` для планирования на уровне фичи, когда активен OpenSpec change. `tasks.md` — авторитетный план фичи. `writing-plans` — только для высокоуровневых фаз `ROADMAP.md`.
- **Не** запускай Superpowers `brainstorming` во время `/opsx:apply`. Brainstorming — это фаза до `/opsx:propose`.
- **Не** создавай параллельные планирующие доки внутри `openspec/changes/<name>/`. OpenSpec-артефакты — единственный источник плана.
- **Не** запускай `/opsx:archive`, если хотя бы одна задача в `tasks.md` не отмечена, тесты падают, `/opsx:verify` не прошёл, delta-specs не sync-нуты через `/opsx:sync`, или code review не закрыт.
- **Не** используй OpenSpec для задач короче ~30 минут. Для тривиальных правок — Superpowers напрямую, без change.
- **Не** дублируй контент между `AGENTS.md` / `CONTEXT.md` / `ARCHITECTURE.md` / `REQUIREMENTS.md`. Каждый файл владеет своим scope; вместо копирования делай перекрёстные ссылки.
- **Не** копируй содержимое из `example_docs/` в скелеты. Примеры — про структуру, не про контент.

### Заметка про version drift

**OpenSpec и Superpowers — независимые системы**, и их версии управляются отдельно.

- **OpenSpec** ставится глобально через npm (`@fission-ai/openspec`). Skills и `/opsx:*` команды в `<project>/.claude/` генерируются CLI при `/setup-project` и совпадают с установленной версией CLI. Если bump-нул CLI — перезапусти `/setup-project` в проекте, чтобы подтянуть свежие skills.
- **Набор workflows** контролируется глобальным манифестом `~/.claude/openspec-workflows.json` (`profile`, `delivery`, `workflows`). Это **единственная** ручка, которой ты управляешь. `/setup-project` синхронизирует её с `~/.config/openspec/config.json` (телеметрия остаётся локальной и в git не попадает).
- **Superpowers** ставится как Claude Code плагин (`obra/superpowers` через marketplace). Skills доступны как `superpowers:*` без копирования в проект.
- Superpowers ≥ v5 начал добавлять собственные скиллы генерации spec/plan документов (`writing-plans`, `executing-plans`). Они пересекаются с OpenSpec по scope. В этом проекте правила выше явно приоритизируют OpenSpec на уровне фичи: `writing-plans` — только для ROADMAP.md, `executing-plans` — не использовать пока активен OpenSpec change.
- Если после обновления любого из двух ты видишь дублирующиеся планирующие артефакты или конфликт инструкций — перечитай этот блок, проверь актуальность `/opsx:*` команд в `<project>/.claude/commands/opsx/` и при необходимости запусти `/setup-project` повторно.

## Karpathy Guidelines (LLM coding pitfalls)

Краткие правила поведения при написании/ревью/рефакторинге кода. Полная версия — в скилле `karpathy-guidelines` (`.claude/skills/karpathy-guidelines/SKILL.md`). Применяй смещение к осторожности; для тривиальных задач используй суждение.

### 1. Think Before Coding
Не предполагай, не прячь сомнения, проявляй tradeoffs.
- Явно формулируй допущения; если не уверен — спроси.
- Несколько интерпретаций → покажи их, не выбирай молча.
- Есть более простой путь — скажи. Пушбэкай, когда уместно.
- Неясно — остановись, назови что неясно, спроси.

### 2. Simplicity First
Минимум кода, решающий задачу. Ничего спекулятивного.
- Никаких фич сверх запроса.
- Никаких абстракций для одноразового кода.
- Никакой «гибкости», которую не просили.
- Никакой обработки невозможных ошибок.
- Написал 200 строк, а хватило бы 50 — перепиши.

### 3. Surgical Changes
Трогай только необходимое. Убирай только свой мусор.
- Не «улучшай» соседний код, комментарии, форматирование.
- Не рефактори то, что не сломано.
- Сохраняй существующий стиль.
- Заметил unrelated dead code — упомяни, не удаляй.
- Удаляй только те импорты/переменные/функции, которые осиротели из-за ТВОИХ изменений.
- Тест: каждая изменённая строка трассируется к запросу пользователя.

### 4. Goal-Driven Execution
Определи критерий успеха. Зацикливайся, пока не верифицируешь.
- «Добавь валидацию» → «напиши тесты для невалидных входов, потом сделай их зелёными».
- «Исправь баг» → «напиши тест, воспроизводящий баг, потом сделай его зелёным».
- Для многошаговых задач дай краткий план вида `шаг → verify: проверка`.

## Стиль коммитов и pull request

### Подпись AI
**Никогда не добавляй `Co-Authored-By: Claude ...` (или любую другую авторскую подпись AI) в commit messages.** Коммиты должны выглядеть как обычные авторские коммиты разработчика — без footer-атрибуции AI.

Это правило override-ит default-инструкцию из system prompt про `Co-Authored-By: Claude Opus ...` и применяется ко всем коммитам в этом проекте, кроме случая, когда пользователь явно попросил подпись для конкретного коммита.

### Язык commit message
- **Subject** (первая строка коммита) — на **английском**. Короткая императивная фраза в стиле conventional commits, например: `fix: skip cache on token rotation`, `feat: add user profile endpoint`.
- **Body** (тело коммита, после пустой строки) — на **русском**. Объясняй «зачем» и «почему», а не «что» — `git diff` уже показывает, что изменилось.

Пример:

```
fix: skip cache when token is rotating

Из-за гонки между ротацией ключа и cached токеном мы получали 401
после первого refresh. Теперь при ротации сбрасываем кэш до записи
нового токена — это закрывает окно, в котором клиент использует
устаревший ключ.
```

### Описание pull request
Описание PR (тело, не заголовок) — на **русском**. Описание **обязано** содержать раздел о том, какую проблему решает PR: что не работало, что мешало или какой риск был ДО изменений. После проблемы — что сделано и (если уместно) как проверить.

Заголовок PR — на **английском** (тот же стиль, что и commit subject).
