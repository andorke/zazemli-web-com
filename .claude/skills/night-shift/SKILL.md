# Night-shift: команды дня и утра

Спека: `~/.claude/docs/specs/2026-07-10-night-shift-design.md`. Раннер и гейты — bash (`scripts/`), эта инструкция — только про интерактивные операции. Если в проекте есть копия `.claude/skills/night-shift/` — работай с НЕЙ (project copy wins), глобальный шаблон не трогай.

## list — нумерованный список changes (шаг 1 вечернего флоу)

1. Просканируй `openspec/changes/*/` (без `archive/`): кандидаты — папки, где в `tasks.md` есть строки `- [ ]`.
2. Для каждого прочитай `proposal.md` (и при неясности `design.md`) и составь суммаризацию В ОДНУ СТРОКУ: что делает change и что осталось. Достань git-даты папки change: создан = `git log --reverse --format=%as -- openspec/changes/<имя> | head -1`, обновлён = `git log -1 --format=%as -- openspec/changes/<имя>` (плюс дней с последнего изменения). Первая незакрытая `- [ ]` из tasks.md = «следующая задача» (обрежь до ~60 символов). Если существует `.night/state/<имя>/` — пометь 🌙 (ночь уже пробовала; загляни в PROGRESS.md на ESCALATE).
3. Отсортируй кандидатов по дате «обновлён» по убыванию (свежие сверху; при равенстве — по имени) и выведи нумерованную таблицу: `№ | change | задач [x]/всего | создан | обновлён (дней) | следующая задача | о чём`. Номера присваиваются ПОСЛЕ сортировки — и в таблице, и в last-list.json порядок одинаковый. Отметь риски ночепригодности (неполные артефакты, пересечение с forbidden_paths, свежий WIP в той же зоне) значком ⚠️ с пояснением после таблицы; 🌙 — были ночные попытки.
4. Сохрани маппинг номеров АТОМАРНО в `.night/last-list.json`: `{"created":"<дата-время>","items":[{"n":1,"change":"<имя>"},...]}` (mkdir -p .night; mktemp+mv). Номера из этого файла использует команда start — не полагайся на память диалога.
5. Заверши подсказкой: «Запуск: \"запусти ночь HH:MM-HH:MM с changes N,M[,K=модель]\"».

## start — запустить ночную разработку (шаг 2: время + список)

Вход от пользователя: начало и конец ночи (HH:MM-HH:MM) и список changes — номера из последнего list и/или имена; опционально модель per-change («3=sonnet»), max_episodes, база ветвления («от <ветка>» / «from <ветка>») и ветка вливания MR («в <ветка>» / «into <ветка>»).

1. Разреши номера через `.night/last-list.json` (файла нет или номер отсутствует → выполни list заново и попроси подтвердить). Для каждого change проверь: папка существует, в tasks.md есть `- [ ]`.
2. Обнови `night.config.json` (jq, mktemp+mv): `.schedule.start`/`.schedule.end` из аргументов (валидация HH:MM, start != end); указана база → `.base_ref = "<ветка>"` (проверь, что реф существует: `git rev-parse --verify <ветка>` или `origin/<ветка>`; локальная незапушенная база допустима — ночные ветки создаются от неё и пушатся сами); указана ветка вливания → `.mr.target_branch = "<ветка>"` (эта ДОЛЖНА существовать на origin, иначе MR не создастся). База/target не указаны → соответствующие поля не трогай.
3. Запиши очередь `.night/queue.json` (mktemp+mv): `{"run_id":"night-<YYYYMMDD-HHMMSS текущего времени>","state":"prepared","date":"<сегодня>","items":[{"change":..., "model":"<явная или пусто=дефолт конфига>", "max_episodes":<из конфига или явный>, "status":"queued", "episodes_done":0, "note":"", "mr_url":""}]}` — порядок = порядок пользователя.
4. Перегенерируй launchd-джобу под новое время: plist `~/Library/LaunchAgents/com.night-shift.<имя-папки-проекта>.plist` существует → `bash .claude/skills/night-shift/scripts/install-launchd.sh --keep-tokens` (токены подтянутся из старого plist, интерактива не будет). Plist ещё НЕ существует → скажи пользователю один раз запустить в ТЕРМИНАЛЕ: `bash .claude/skills/night-shift/scripts/install-launchd.sh` (интерактивный ввод токенов; из сессии упадёт).
5. Если очередь на Opus суммарно больше ~3 change — предупреди про недельный кап Opus (24–40 ч/нед на Max 20x).
5а. Прогноз вместимости окна (changes идут ПОСЛЕДОВАТЕЛЬНО): суммарные эпизоды очереди (Σ max_episodes) × ~6 мин + ~15 мин на ночное ревью каждого change с работой; если оценка заметно больше окна — предупреди, какие changes в хвосте рискуют получить 0–1 эпизод, и предложи сократить очередь, переставить приоритетные вперёд или расширить окно.
6. Итоговая сводка: «Ночь HH:MM–HH:MM. База: <base_ref или origin/main>. MR → <target_branch>. Очередь: 1) change (модель) …». Проверь `launchctl list | grep night-shift` и покажи статус. Напомни: Mac в питании и залогинен; changes должны быть запушены в origin/main.

## view — какие джобы запущены и их статус

Read-only сводка. Собери и покажи:
1. **Все launchd-джобы night-shift** (по всем проектам): `ls ~/Library/LaunchAgents/com.night-shift.*.plist` → для каждого plist: проект (из имени файла), время старта (`plutil -extract StartCalendarInterval.Hour raw <plist>` + Minute), состояние: `launchctl list | grep com.night-shift` — колонки «PID Status Label» (PID есть = ран работает прямо сейчас; PID `-` = ждёт расписания, Status = код последнего завершения, 0 = штатно). Plist-ов нет — «launchd-джобы не установлены».
2. **Текущий проект**: ПЕРВОЙ строкой — идентичность рана из `.night/queue.json`: `run_id` и `state` (prepared = взведён, ждёт старта; in_progress = идёт; done/failed = завершён, + started_at/finished_at/fail_note). Если state=in_progress, а pid лока мёртв или лока нет — покажи «in_progress (crashed/stale — рекламация при следующем старте)». Затем: живой ран — `.night/run.lock` существует и `kill -0 $(cat .night/run.lock/pid)` (лок есть, pid мёртв → пометь «стейл-лок, будет рекламирован следующим стартом»); статусы очереди таблицей из `.night/queue.json` (status / episodes_done / mr_url; файла нет — «очередь не собрана»); хвост `tail -8 .night/logs/<сегодня>/runner.log` (нет файла — вчерашней датой; вообще нет — «ночь ещё не бегала»); ветки `git branch --list 'night/*'`; следующий старт из `night.config.json` schedule + установлена ли джоба ЭТОГО проекта (пересечение с п.1).
3. Итоговая строка-вердикт одним предложением: «ран идёт (change X, эпизод N)» / «ждёт старта в HH:MM (launchd установлен)» / «сегодня стартовать нечему: нет launchd-джобы / нет очереди».
Ничего не менять и не запускать — только чтение.

## report — утренний разбор
0. Шапка отчёта — идентичность рана: из `.night/queue.json` возьми `run_id` (внутри — дата-время старта: `night-YYYYMMDD-HHMMSS`), `state`, `started_at`/`finished_at`; окно/база/target — из night.config.json. Формат: «Ран `night-…` (старт 11.07 20:23, финиш 21:40): база <base_ref> → MR в <target>». Если runner.log за дату содержит несколько стартов — разбирай ран из queue.json (последний); прежние попытки упомяни по строке каждая (время + причина FATAL).
1. Прочитай `.night/NIGHT_REPORT.md`, затем по каждому change: `state/<change>/REVIEW.md`, `PROGRESS.md` (ESCALATE/REJECTED), при нужде — логи `.night/logs/<date>/`. Для failed-change'ей первым источником открывай `.night/logs/<date>/runner.log` — хронологию оркестратора (старт/эпизоды/rc/ожидания лимитов).
2. Дай разбор по трём корзинам: мержить (done, чистое ревью) / добить днём (needs_review — что именно чинить) / вернуть в ночь или закрыть (failed — почему упало). Статус `ready_for_owner` — отдельной строкой при корзинах: агентская часть закрыта, дневной кодинг не нужен — сразу в смоук-чеклист владельца (п.4). Отдельно подсвети эскалации — это то, что sensors не ловят (неверный диагноз, оверинжиниринг, непонятая спека).
2а. Напротив каждого сделанного change (done и needs_review) давай пару «Было → Стало»: «было» — симптом/поведение до изменения, одной-двумя фразами из proposal.md (Why); «стало» — новое поведение по факту ночной реализации (из REVIEW.md и диффа ночной ветки — что реально изменилось, включая частичность), а не пересказ плана.
2б. Обязательно приведи из NIGHT_REPORT.md секцию «Сессии Claude»: ПОЛНЫЙ session_id каждого ночного вызова (UUID целиком, никогда не сокращай и не обрезай) с готовой к копированию командой `claude --resume <полный-id>` — владелец может зайти в любую сессию и посмотреть, как выполнялась задача, или продолжить её. Если секции нет (старый отчёт) — собери сам: `jq -r '.session_id // empty' .night/logs/<дата>/*/*.json`.
3. Ссылки на MR — из отчёта. После мержа напомни про `/opsx:verify` и `/opsx:archive`.
4. Заверши отчёт секцией «Что просмоучить»: по каждому change со статусом done/needs_review собери незакрытые `- [ ]`-пункты ручной проверки (smoke, «ручная проверка», on-device) из tasks.md НОЧНОЙ ветки — `git show night/<change>:openspec/changes/<change>/tasks.md` (ночь отмечает [x] сделанное, tasks.md текущей ветки устарел; рефа нет — бери локальный tasks.md). Выведи единый чек-лист, сгруппированный по change'ам: пункт + краткие шаги прогона (установка, сценарий, ожидаемый результат). Ночь ручные проверки не закрывает (verify = сборка) — этот чек-лист и есть критерий приёмки реализации перед мержем.
5. Полный разбор оформи HTML-отчётом скиллом visual-explainer (Skill tool): одна страница; шапка — идентичность рана из п.0 (run_id, старт/финиш, база → target); секции «Сводка ночи» (очередь со статусами и эпизодами), «Корзины» (мержить / добить днём / вернуть или закрыть — с причинами), «Changes» — по каждому change раскрывающаяся карточка (`<details>`) с drill-down «что конкретно сделано»: «Было → Стало», коммиты ночной ветки (`git log --oneline <база>..night/<change>`), изменённые файлы (`git diff --stat <база>..night/<change>`), закрытые за ночь пункты tasks.md, находки ревью, сессии change (полные session_id + копируемая команда `claude --resume <полный-id>`), MR; далее «Эскалации», «Сессии Claude» (полный список вызовов: эпизоды/ревью/фиксы, id целиком в моноширинных копируемых блоках), «MR» (ссылки), «Что просмоучить» (чек-лист из п.4). Сохрани страницу в `.night/NIGHT_REPORT.html` и открой в браузере; в чат — краткую выжимку (3–5 строк) с путём к файлу. Скилла visual-explainer нет — весь разбор текстом в чат, как раньше.

## install — установить в текущий проект
1. Убедись, что это git-репо с openspec/. 2. Скопируй шаблон: `rsync -a --ignore-existing ~/.claude/skills/night-shift/ .claude/skills/night-shift/` — сначала прогони с `--dry-run -v`, покажи пользователю новые/конфликтные файлы; НЕ перезаписывай существующие файлы проекта без подтверждения (спроси при конфликте). 3. Сгенерируй night.config.json из templates/night.config.json интервью: verify-команда, lint, модели, schedule, mr.host/project, forbidden_paths. 4. Допиши `.night/` в .gitignore. 5. Добавь в AGENTS.md секцию «Ночной флоу» (очередь из opsx-changes; спеки ночью read-only; статусы и отчёты в .night/). 6. Предложи адаптировать prompts/ копии под проект. 7. Предложи `scripts/install-launchd.sh`. 8. install-launchd.sh запускается владельцем в терминале (интерактивный ввод токенов; из сессии read -s упадёт).

## upgrade — обновить копию из шаблона
1. Сравни версии: `TEMPLATE_VERSION` копии vs `~/.claude/skills/night-shift/TEMPLATE_VERSION`. Равны — скажи «актуально». 2. Покажи `diff -ru` по scripts/ prompts/ templates/ (копия vs шаблон). 3. Предложи перенести обновления, СОХРАНЯЯ проектные правки (project copy wins) — конфликтные куски показывай и спрашивай. 4. Обнови TEMPLATE_VERSION копии.

Правило сопровождения шаблона: ЛЮБАЯ правка файлов глобального шаблона обязана сопровождаться бампом TEMPLATE_VERSION — иначе upgrade у проектных копий скажет «актуально» и обновление не доедет никогда.

## Правила безопасности
- Никогда не запускай night-run.sh без --now из сессии (его запускает launchd).
- Не редактируй runtime-поля queue.json вручную во время идущего рана (есть .night/run.lock).
- Токены живут только в plist — не выводи их в чат и не коммить.
